     1	# go-go-goja – Native-Module Playground for goja
     2	
     3	**go-go-goja** is a tiny sandbox project that shows how to wire Go-implemented "native" modules into a [dop251/goja] JavaScript runtime using the [goja_nodejs/require] subsystem.
     4	
     5	The goal is to have a place where you can:
     6	
     7	* add your own Go files under `modules/` and immediately use them from JS via `require("your-module")`
     8	* experiment with goja + Node-style `require()` without having to set up a whole application
     9	* open the canonical interactive prompt with `go run ./cmd/goja-repl tui`
    10	* inspect and persist REPL sessions through the `goja-repl` CLI and JSON server surfaces
    11	* compose runtime behavior explicitly via `engine.NewBuilder() -> Build() -> Factory.NewRuntime(...)`
    12	
    13	---
    14	
    15	## Folder layout
    16	
    17	```text
    18	 go-go-goja/
    19	 ├── cmd/
    20	 │   ├── goja-repl/       # canonical REPL CLI, TUI, and JSON server
    21	 │   └── bun-demo/        # bun-integrated demo command
    22	 ├── engine/              # builder/factory/runtime ownership APIs
    23	 ├── modules/             # ← add your Go-backed modules here
    24	 │   ├── common.go        # registry plumbing (NativeModule, Register, …)
    25	 │   ├── events/          # Go-native Node-style EventEmitter
    26	 │   ├── fs/              # example module 1: basic file-system helpers
    27	 │   ├── exec/            # example module 2: thin wrapper around os/exec
    28	 │   ├── yaml/            # example module 3: YAML parse/stringify/validate
    29	 ├── testdata/            # demo JS scripts used by Go tests
    30	 ├── repl_test.go         # Go test that runs a JS script through the runner
    31	 └── go.mod
    32	```
    33	
    34	The canonical API is explicit runtime composition:
    35	
    36	1. create a builder (`engine.NewBuilder(...)`)
    37	2. add module/runtime options (`WithModules(...)`, `WithRuntimeInitializers(...)`, ...)
    38	3. build an immutable factory (`Build()`)
    39	4. create an owned runtime (`factory.NewRuntime(ctx)`)
    40	5. close runtime explicitly (`rt.Close(ctx)`)
    41	
    42	Legacy convenience wrappers (`engine.New()`, `engine.NewWithOptions(...)`, `engine.Open(...)`) were removed.
    43	
    44	---

    70	### Module Security Flags
    71	
    72	You can control which native modules are loaded using persistent CLI flags:
    73	
    74	```bash
    75	# Safe mode: only data-only modules (crypto, events, path, time, timer)
    76	go run ./cmd/goja-repl --safe-mode run ./script.js
    77	
    78	# Whitelist: only specific modules
    79	go run ./cmd/goja-repl --enable-module fs,path run ./script.js
    80	
    81	# Whitelist in the TUI; db is an alias for the database module
    82	go run ./cmd/goja-repl tui --enable-module db
    83	
    84	# Blacklist: all except specific modules
    85	go run ./cmd/goja-repl --disable-module fs,exec run ./script.js
    86	```
    87	
    88	These flags apply to all commands that build a runtime (`run`, `tui`, `eval`, `create`, etc.).

    36	## Architecture Overview
    37	
    38	In the Turn-based architecture:
    39	- **Provider engines** emit `tool_call` blocks when models request tools
    40	- **The tool loop runner** executes tools and appends `tool_use` blocks
    41	- **Engines re-run** with the updated Turn to let the model continue
    42	
    43	> **Key Pattern:** The runtime `tools.ToolRegistry` is carried via `context.Context` (see `tools.WithRegistry`). Only serializable tool configuration lives on `Turn.Data` (e.g., `engine.KeyToolConfig`). This keeps Turn state persistable while allowing dynamic tools per inference call.

    31	## Why `glaze help export` exists
    32	
    33	The Glazed help system stores documentation in an SQLite-backed store at runtime. That makes querying fast, but it also means the data is trapped inside a running process. `glaze help export` solves this by letting you dump the entire help tree—or a filtered slice of it—to formats you can use elsewhere.
    34	
    35	This matters when you want to:
    36	
    37	- back up or version-control your help content as plain files,
    38	- feed help metadata into an external search index or CMS,
    39	- generate reports on documentation coverage (which commands have examples, which topics are missing),
    40	- or ship a SQLite database of your app's documentation alongside the binary.
    41	
    42	The command is built on the same `HelpSystem` and `Store` primitives that `glaze serve` and `glaze render-site` use, so the exported data is always consistent with what the CLI and the web browser show.
    43	
    44	## Basic usage
    45	
    46	The simplest invocation exports every loaded section as JSON:
    47	
    48	```bash
    49	glaze help export --output json
    50	```
    51	
    52	By default, `--with-content` is `true`, so the JSON includes the full markdown body of each section. If you only need metadata, disable content:
    53	
    54	```bash
    55	glaze help export --with-content=false --output csv
    56	```
    57	
    58	## Export formats
    59	
    60	The `--format` flag controls the export target:
    61	
    62	| Format | What it produces | Best for |
    63	|--------|-----------------|----------|
    64	| `glazed` (default) | Tabular output via the GlazeProcessor | JSON, CSV, YAML, or table dumps to stdout |
    65	| `files` | One `.md` file per section on disk | Backup, version control, or editing |
    66	| `sqlite` | A standalone SQLite database | Querying with SQL, shipping with apps |
    67	
    68	### Tabular export (`--format glazed`)
    69	
    70	This is the default. It behaves like any other Glazed command: you can choose the output serializer with `--output`.
    71	
    72	```bash
    73	# Pretty-printed JSON with full content
    74	glaze help export --output json
    75	
    76	# Compact CSV for spreadsheet import
    77	glaze help export --with-content=false --output csv
    78	
    79	# YAML for human review
    80	glaze help export --output yaml
    81	```
    82	
    83	The tabular mode emits one row per section with these columns:
    84	
    85	| Column | Description |
    86	|--------|-------------|
    87	| `slug` | Unique section identifier |
    88	| `title` | Display title |
    89	| `short` | One-line description |
    90	| `content` | Full markdown body (omitted when `--with-content=false`) |
    91	| `section_type` | `GeneralTopic`, `Example`, `Application`, or `Tutorial` |
    92	| `topics` | Comma-separated topic tags |
    93	| `commands` | Comma-linked command associations |
    94	| `flags` | Comma-linked flag associations |
    95	| `is_top_level` | Whether the section appears in top-level listings |

    31	## Overview
    32	
    33	The Glazed help system provides a structured, queryable approach to CLI documentation that goes beyond basic command help. It organizes documentation into typed sections (topics, examples, applications, tutorials) with rich metadata for filtering and discovery. The system supports both human-readable help pages and programmatic querying through a simple DSL, making it easy to build comprehensive CLI documentation that users can explore efficiently.
    34	
    35	The help system stores sections in an SQLite-backed store, enabling fast queries, text search, and metadata filtering. You typically load documentation from markdown files with YAML frontmatter at startup, creating a self-contained help database that you can query from both the command line and Go code.
    36	
    37	You can also export the entire help tree—or a filtered slice of it—to JSON, CSV, YAML, individual markdown files, or a standalone SQLite database. This is useful for backups, external indexing, and shipping documentation alongside your application. See [`glaze help export-help-entries`](glaze help export-help-entries) for details.
    38	
    39	## Section Types and Structure
    40	
    41	Help sections follow a type-based classification system that separates conceptual documentation from practical examples. This separation enables precise filtering and contextual help display, allowing users to find exactly the type of information they need based on their current task.
    42	
    43	### Section Types
    44	
    45	- **GeneralTopic**: Conceptual documentation explaining how features work
    46	- **Example**: Focused demonstrations of specific command usage
    47	- **Application**: Real-world use cases combining multiple features
    48	- **Tutorial**: Step-by-step guides for complex workflows
    49

    73	    Order          int  // Sort order within type
    74	}
